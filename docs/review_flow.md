# 設計・実装成果物のレビューフロー

他エージェントが作成した資料やコードを、**レビューエージェントが自動で読み取り、レビュー → ユーザー確認 → 作成者エージェントが対応**する流れです。チャットへの貼り付けは不要です。

---

## 1. フロー概要

1. **成果物の作成**  
   architect_agent または engineer_agent が設計書・コードを作成し、保存先パス（例: `docs/system_dev/er_diagram_xxx.md`）を Coordinator に報告する。

2. **自動レビュー依頼**  
   Coordinator がその**パス**を review_agent に渡してレビューを依頼する（例:「以下のファイルをレビューしてください。パス: docs/system_dev/er_diagram_xxx.md」）。

3. **レビュー実行**  
   review_agent が **read_file_for_review** ツールで該当ファイルの内容を読み取り、機密・型・脆弱性・動作などの観点でレビューする。フォーマットを崩さずにファイルを参照できる。

4. **ユーザー確認**  
   レビュー結果（要約・指摘一覧・重要度・推奨アクション）をユーザーに提示する。

5. **レビュー対応**  
   ユーザーが対応を求めた場合、Coordinator が**成果物を作成したエージェント**（architect_agent または engineer_agent）にレビュー内容を伝え、修正を依頼する。

---

## 2. review_agent のツール

| ツール | 説明 |
|--------|------|
| **read_file_for_review(relative_path)** | プロジェクトルートからの相対パスでファイルを読み取り、内容を返す。他エージェントが作成した資料・コードのパスが渡されたときに使用する。 |

- パスは **プロジェクトルートからの相対パス**（例: `docs/system_dev/user_manual_README.md`）。  
- プロジェクトルートは環境変数 `OUTPUT_PROJECT_ROOT` または `PROJECT_ROOT` で変更可能（未設定時は A4A ルート）。  
- パストラバーサル対策のため、プロジェクト外のパスは拒否される。

---

## 3. Coordinator の役割

- architect_agent / engineer_agent から**保存先パス**の報告を受けたら、そのパスを review_agent に渡してレビューを依頼する。  
- レビュー結果をユーザーに提示し、確認してもらう。  
- レビュー対応が必要な場合、作成者エージェントにレビュー内容を伝えて修正を依頼する。

---

## 4. 運用上の注意

- **Coordinator 経由**で依頼する（Web UI は 1 エージェント直行のため、この一連のフローは Coordinator に話しかけるときに有効）。  
- 各エージェントを別ターミナルで起動している場合は、Coordinator（8000）にだけクエリを送る。
