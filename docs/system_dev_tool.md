# システム開発ツール（A4A エージェント構成）

A4A 上で動かすシステム開発用のエージェント群です。PM がユーザーと他エージェントの仲介をし、設計・レビュー・実装・運用・企画を分担します。

## エージェント一覧

| 役割 | エージェント名 | 説明 |
|------|----------------|------|
| **PM** | coordinator_agent (8000) | ユーザーと他エージェントの仲介。命令受付・進捗報告。 |
| **アーキテクト** | architect_agent | PM の指示で設計。フロー図・ER図・ユーザーマニュアル・画面項目定義書・画面遷移図を `docs/system_dev/` に格納。 |
| **レビュー** | review_agent | 他エージェントの暴走防止。PMO/QA 的・否定的立場。機密・型・脆弱性・動作・エラーをチェック。結果を PM・クライアントに共有。 |
| **エンジニア** | engineer_agent | アーキテクトの指示で実装。レビュー指摘で修正。 |
| **運用保守** | ops_agent | メトリクス、異常時 LOG、原因・影響・対応方針、クライアント提案、承認後自動修正。 |
| **企画** | planning_agent | システムの可能性・課題、次機能の候補と費用対効果・実現性・工期の提案。 |
| **企画B** | planning_b_agent | planning_agent の提案に否定的立場で検証。多角的な判断材料を提供。 |

## 使い方

```bash
make run
```

別ターミナルでユーザーは **8000 番（PM）** に話しかけます。

```bash
make query q="〇〇システムの画面遷移図と画面項目定義書を設計して"
make query q="今の設計をレビューして"
make query q="次のリリースで入れる機能の候補と費用対効果を企画して。その上で企画Bの反論も聞かせて"
```

## 設計成果物の格納先

- アーキテクトが `write_design_doc` で出力する先: **`docs/system_dev/`**
  - フロー図、ER図、ユーザーマニュアル、画面項目定義書、画面遷移図（いずれも Markdown 等）

## レビュー・企画の流れ

- **レビュー**: review_agent が指摘 → PM がクライアントに共有 → クライアント要望に合わせて engineer 等が「受け入れる」か「説き伏せる」かを選択。
- **企画**: planning_agent が提案 → planning_b_agent が否定的観点で検証 → PM が両方をまとめてユーザーに提示。
