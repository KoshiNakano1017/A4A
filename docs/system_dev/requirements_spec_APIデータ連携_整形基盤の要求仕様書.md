# 要求仕様書：APIデータ連携・整形基盤

## 1. 目的
本プロジェクトは、システム間およびAIエージェントと外部API間のデータ連携において発生する「フォーマット不一致」を解消するための、標準的かつ保守性の高いデータ整形基盤を構築することを目的とする。

## 2. 背景
現状、接続先APIごとにレスポンス形式が異なり、AIエージェントや後続システムでのデータ処理に不整合やエラーが発生している。これを柔軟に吸収しつつ、長期的な運用コスト（技術負債）を抑制する仕組みが求められている。

## 3. 基本方針
「ローコードによる可視性と標準化」を優先し、複雑な処理のみ「カスタムコード」で補完するハイブリッドアプローチを採用する。

- **原則（Hub層）**: Google Cloud **Application Integration** を中心に据え、GUIベースでデータマッピングとフローの可視化を行う。
- **例外（カスタム層）**: 標準機能で実装が困難な複雑なアルゴリズム、高度なバリデーション、特殊な暗号化等は **Cloud Functions (2nd gen)** をコンポーネントとして呼び出す。

## 4. 機能要件

### 4.1 データ整形機能
- 異なるスキーマ（JSON, XML, CSV等）間の相互変換。
- ターゲットシステム（AIエージェント等）が期待するフォーマットへのマッピング。
- メタデータの付与（タイムスタンプ、リクエストID等）。

### 4.2 連携・オーケストレーション
- 複数のAPIを順次実行し、中間データを加工して次ステップへ渡す機能。
- エラー発生時のリトライ制御およびフォールバック処理。

### 4.3 監視・ガバナンス
- 実行ログの集中管理（Cloud Logging連携）。
- GUI上でのデータフローの可視化。
- 接続先APIの認証情報の安全な管理（Secret Manager連携）。

## 5. 非機能要件

### 5.1 保守性
- データマッピングの変更は、可能な限りコードの修正なしで（Application Integrationの設定変更のみで）完結させること。
- カスタムコード（Cloud Functions）を利用する場合は、ランタイムのEOL対応を考慮した設計とすること。

### 5.2 パフォーマンス
- リアルタイム性が求められる連携では、Cloud Functionsのコールドスタート対策（最小インスタンス数の設定等）を行うこと。

### 5.3 セキュリティ
- 全ての通信はIAMによる適切な認証・認可を行い、最小権限の原則に従うこと。

## 6. 成功基準
1. 異なるAPIからのレスポンスが、定義されたターゲットスキーマに正しく変換されること。
2. データ連携のフローがGoogle Cloudコンソール上で視覚的に把握できること。
3. 新規APIの追加時、開発・デプロイ工数が従来（フルスクラッチ）比で削減されること。
