# 要求仕様書：APIデータ連携・整形基盤（ハイブリッド方式）

## 1. 背景と目的
現在、外部APIおよび内部システム間、ならびにAIエージェントが利用するデータにおいて、フォーマットの不一致が発生している。これにより、AIエージェントの推論精度低下やシステム連携の複雑化を招いている。
本プロジェクトでは、Google Cloudのマネージドサービスを活用し、保守性と柔軟性を両立した「翻訳機」としてのデータ連携基盤を構築する。

## 2. 基本方針
「運用持続性」を最優先事項とし、以下のハイブリッド構成を採用する。
- **原則（標準）**: `Application Integration` をデータ連携のハブ（バス）として採用し、GUIベースのデータマッピングによって可視性と保守性を確保する。
- **例外（カスタム）**: `Application Integration` の標準機能では実装が困難な「複雑なビジネスロジック」や「高度な正規表現・計算」が必要な場合のみ、`Cloud Functions (2nd gen)` を部品として呼び出す。

## 3. 機能要件
### 3.1 データ連携・変換機能
- **入力受領**: 複数の外部API（JSON/XML等）からのリクエスト/レスポンスをトリガーとして受領する。
- **データマッピング**: `Application Integration` の Data Mapping タスクを用い、出力先（AIエージェントや他システム）が期待するスキーマへ変換する。
- **カスタム整形**:
    - 複雑なデータ変換が必要な場合、Python等の言語を用いた `Cloud Functions` を実行する。
    - Cloud Functions は Application Integration からコネクタ経由で呼び出される。
- **出力提供**: 整形済みデータを、ターゲットとなるシステムやAIエージェントへ返却する。

### 3.2 運用・管理機能
- **可視化**: データの流れ（フロー）を `Application Integration` のデザイナー画面で確認可能とする。
- **エラーハンドリング**: 変換失敗時、またはAPI呼出失敗時のエラー内容を捕捉し、ログ記録および通知を行う。
- **ロギング**: `Cloud Logging` と連携し、リクエスト単位での追跡（トレース）を可能にする。

## 4. 非機能要件
### 4.1 保守性
- カスタムコード（Cloud Functions）の利用は、標準機能で代替不可な場合に限定する（技術負債の抑制）。
- Cloud Functions のランタイムは最新の長期サポート版を使用し、CI/CD環境による自動テスト・デプロイを行う。

### 4.2 セキュリティ
- **認証・認可**: 各コンポーネント間の通信は IAM（Identity and Access Management）により最小権限で認可する。
- **API保護**: 外部連携が必要な場合は必要に応じて `API Gateway` 等の検討を行う。

### 4.3 パフォーマンス
- `Cloud Functions` を使用する場合、コールドスタートによる遅延を考慮し、要件に応じて `min-instances` を設定する。

## 5. 将来の拡張性
- 将来的に連携先がSaaS（Salesforce等）へ拡大する場合、`Integration Connectors` を追加することで容易に拡張可能とする。
