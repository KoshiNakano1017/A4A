# テスト仕様書: システム開発ツール 動作確認

## 目的

- システム開発まわりのエージェント（PM・アーキテクト・レビュー・エンジニア・運用・企画・企画B・セキュリティオフィサー・プロンプトエンジニア）が意図どおり動作することを確認する。
- **リアルタイムで各エージェントの動きをモニタリングしながら**テストを実施する方法を含む。

## 前提条件

- プロジェクトルートで `uv sync` 済み
- `.env` に `GEMINI_API_KEY` 設定済み
- 手順の詳細は `docs/verification_procedure.md`、モニタリングの詳細は `docs/agent_monitoring.md` を参照

---

## リアルタイムモニタリングの方法（テスト実施時は必須）

テスト中に「どのエージェントが何をしているか」を確認するため、**ログ付きで一括起動**する。

### 手順

1. **ターミナル1** でプロジェクトルートから起動する。
   ```bash
   make run-verbose
   ```
   Windows で `make` が使えない場合:
   ```powershell
   python -m a4a.run_all --verbose
   ```

2. 起動ログで次を確認する。
   - 「Discovering agents...」のあと、**Started ○○ on port ○○○○** が並ぶこと。
   - **coordinator_agent** が最後に **port 8000** で起動していること。
   - 途中で **Process ... exited with code 1** やトレースバックが出ていないこと。

3. テスト用クエリは **ターミナル2** で送る。
   ```bash
   make query q="ここにクエリ"
   ```
   Windows:
   ```powershell
   python -m a4a.a2a_query "ここにクエリ" --port 8000
   ```

4. **ターミナル1** のログで、クエリ送信ごとに次のように表示される。
   - `[coordinator_agent:8000]` … PM がリクエストを受信
   - `[architect_agent:8XXX]` … 設計を依頼されたとき
   - `[review_agent:8XXX]` … レビューを依頼されたとき
   - など、**依頼されたエージェントのプレフィックス**で `POST /a2a/... 200` が増える。

**ポイント**: シナリオごとに「このクエリでこのエージェントが動くはず」を決めておき、ターミナル1でその `[エージェント名:ポート]` のログが出るかで正しくルーティングされているかを確認する。

## エージェント別テストシナリオ

### 1. coordinator_agent（PM）

| 項目 | 内容 |
|------|------|
| 役割 | ユーザーと他エージェントの仲介。依頼の具体化・ルーティング・報告。 |
| テストクエリ | `こんにちは。あなたの役割を一言で教えてください` |
| 期待結果 | エラーにならず、PM としての役割説明が返る。ターミナル1に `[coordinator_agent:8000]` のログが出る。 |
| モニタリング | ポート 8000 のログで受信・応答が流れること。 |

---

### 2. planning_agent（企画）

| 項目 | 内容 |
|------|------|
| 役割 | システムの可能性・課題を整理し、次に作る機能の候補と費用対効果・実現性・工期を提案。 |
| テストクエリ | `業務を効率化したい。何から手をつけるべきか企画の視点で提案して` |
| 期待結果 | 企画エージェントが相談され、課題整理・候補・評価観点が含まれた応答が返る。`[planning_agent:8XXX]` のログが出る。 |
| モニタリング | ふわっとした要望のため、PM が企画に相談する流れになり、planning_agent のプレフィックスが出現すること。 |

---

### 3. planning_b_agent（企画B・否定的検証）

| 項目 | 内容 |
|------|------|
| 役割 | 企画の提案に否定的立場で検証。リスク・反論・代替案を提示。 |
| テストクエリ | 先に企画相談（上記）を行ったあと、`その企画案について、リスクや反対意見の観点でも検証して` など。 |
| 期待結果 | planning_b_agent が呼ばれ、企画案への批判・リスク・代替視点が含まれた応答が返る。`[planning_b_agent:8XXX]` のログが出る。 |
| モニタリング | 企画と対になる形で企画Bのプレフィックスが出現すること。 |

---

### 4. architect_agent（設計）

| 項目 | 内容 |
|------|------|
| 役割 | フロー図・ER図・画面項目定義書・画面遷移図などを設計し、`docs/system_dev/` に格納。 |
| テストクエリ | `ログイン画面の画面項目定義書と画面遷移図を設計し、成果物は docs に保存して` |
| 期待結果 | architect が依頼され、`docs/system_dev/` に該当する .md が作成される。`[architect_agent:8XXX]` のログが出る。 |
| モニタリング | 設計依頼時に architect_agent のプレフィックスが出現し、続けてファイル作成に関する応答が返ること。 |

---

### 5. engineer_agent（実装）

| 項目 | 内容 |
|------|------|
| 役割 | アーキテクトの設計に沿った実装。レビュー指摘への修正。 |
| テストクエリ | `今ある設計書（docs/system_dev）に基づいて、ログイン処理の実装方針を具体的に書いて` |
| 期待結果 | 設計を参照した実装方針・コードイメージが返る。`[engineer_agent:8XXX]` のログが出る。 |
| モニタリング | 実装依頼時に engineer_agent のプレフィックスが出現すること。 |

---

### 6. review_agent（レビュー・QA的）

| 項目 | 内容 |
|------|------|
| 役割 | 成果物を否定的立場でレビュー。機密・型・脆弱性・動作・エラーをチェック。 |
| テストクエリ | `次のコードをレビューして。問題点を指摘して: def login(user_input): return "SELECT * FROM users WHERE id=" + user_input` |
| 期待結果 | レビューエージェントが呼ばれ、SQLインジェクション等の指摘が含まれた応答が返る。`[review_agent:8XXX]` のログが出る。 |
| モニタリング | レビュー依頼時に review_agent のプレフィックスが出現すること。 |

---

### 7. ops_agent（運用保守）

| 項目 | 内容 |
|------|------|
| 役割 | 稼働メトリクス・異常時のLOG・原因・影響・対応方針の整理。承認後は自動修正の提案。 |
| テストクエリ | `APIのレスポンスが遅い場合、運用としてどうメトリクスとログを取るか、インシデントLOGの書き方の例を出して` |
| 期待結果 | 運用・DevOps的なメトリクス・ログ・インシデントLOGの形式が含まれた応答が返る。`[ops_agent:8XXX]` のログが出る。 |
| モニタリング | 運用相談時に ops_agent のプレフィックスが出現すること。 |

---

### 8. security_officer_agent（セキュリティ・オフィサー）

| 項目 | 内容 |
|------|------|
| 役割 | ADKで生成されるコードの脆弱性をリアルタイムでスキャン。デプロイ前の門番。 |
| テストクエリ | `このコードを脆弱性の観点でチェックして: print(request.GET.get("q"))` または Coordinator 経由で「このコードをセキュリティチェックして」と依頼。 |
| 期待結果 | 脆弱性スキャン的な指摘（XSS・入力検証など）が含まれた応答が返る。`[security_officer_agent:8XXX]` のログが出る。 |
| モニタリング | セキュリティチェック依頼時に security_officer_agent のプレフィックスが出現すること。 |

---

### 9. prompt_engineer_agent（プロンプト・エンジニア）

| 項目 | 内容 |
|------|------|
| 役割 | ターゲットエージェントのシステムプロンプト（instruction）を最適化。LLMに合わせたチューニング。 |
| テストクエリ | `レビューエージェントの instruction を、指摘をより具体的にする方向で改善案を出して` |
| 期待結果 | プロンプト改善案・差分・理由が含まれた応答が返る。`[prompt_engineer_agent:8XXX]` のログが出る。 |
| モニタリング | プロンプト改善依頼時に prompt_engineer_agent のプレフィックスが出現すること。 |

## 個別エージェントの直接テスト（任意）

起動ログでポートを確認し、特定エージェントに直接クエリして応答だけを検証する。

```bash
# 例: review_agent が 8003 のとき
python -m a4a.a2a_query "SELECT * FROM users をそのまま実行するコードの危険性を指摘して" --port 8003
```

- 期待: 該当エージェントから応答が返り、ターミナル1では `[review_agent:8003]` にのみログが出る。
