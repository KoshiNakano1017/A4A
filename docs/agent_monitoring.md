# 各エージェントがリアルタイムで何をしているか確認する方法

A4A で各エージェントの動作をリアルタイムに確認する主な方法をまとめます。

---

## 1. 推奨: verbose モードで一括起動（ログをプレフィックス付きで表示）

各サブエージェントと Coordinator の **標準出力・標準エラー** を、エージェント名とポートのプレフィックス付きで同じターミナルに流します。

### 使い方

```bash
make run-verbose
```

または

```bash
python -m a4a.run_all --verbose
# 短いオプション
python -m a4a.run_all -v
```

### 表示例

```
Discovering agents...
Started architect_agent on port 8001 (verbose)
Started review_agent on port 8002 (verbose)
...
Started coordinator_agent on port 8000 (verbose)

--- 各エージェントのログは [エージェント名:ポート] プレフィックス付きで表示されます。Ctrl+C で終了。---

[architect_agent:8001] INFO:     Started server process [12345]
[architect_agent:8001] INFO:     Uvicorn running on http://0.0.0.0:8001
[coordinator:8000] INFO:     127.0.0.1:54321 - "POST /a2a/... HTTP/1.1" 200
[review_agent:8002] INFO:     127.0.0.1:54322 - "POST /a2a/... HTTP/1.1" 200
```

- **uvicorn** のアクセスログ（どのエンドポイントにリクエストが来たか）が `[エージェント名:ポート]` 付きで出ます。
- Coordinator がどのサブエージェントに転送したかは、どのプレフィックスのログが動いているかで判断できます。

### 注意

- 通常の `make run` ではサブプロセスの出力は接続されないため、ログは見えません。**リアルタイムで見たいときは `run-verbose` を使う**想定です。
- 複数エージェントのログが混ざって出るため、行単位でプレフィックスを見てどのエージェントか判別します。

---

## 2. エージェントごとに別ターミナルで起動する（切り分けして確認）

「どのエージェントが何をしているか」を**完全に分けて**見たい場合は、エージェントごとに別ターミナルで起動します。

**リクエスト／レスポンスのログや作業内容を「ポート指定で起動したターミナル」に出すには、必ず Coordinator（8000）経由でクエリを送ってください。** Web UI（8888）で送るとエージェントは同一プロセス内で動くため、別ターミナルで起動したエージェントにはリクエストが届きません。

### 手順の要点

1. 各エージェントをこれまで通り**別ターミナル**で起動（例: 8014 で planning_agent）
2. **Coordinator を 8000 で起動**（例: `$env:PORT="8000"; uv run python -m a4a.agent`）
3. **クエリは Coordinator に送る**  
   - `make query q="企画エージェント生きているか"`  
   - または `adk web` で http://127.0.0.1:8000 のチャットに入力

各エージェントの作業内容は **Coordinator が応答の末尾で報告**します（`[エージェント活動] エージェント名: 実施したこと, ...` の形式）。同時に、依頼を受けたエージェントのターミナルには「依頼を受信、処理中...」「応答送信開始」などが表示されます。

### ターミナルに表示される活動ログ

各エージェントをポート指定で起動したターミナルでは、そのエージェントが依頼を受けたり応答を返したりしたときに、次のようなメッセージが**同じターミナル**に出力されます。

- **依頼を受信、処理中...** — リクエストを受けた直後
- **応答送信開始** — クライアントへの応答を返し始めたとき
- **architect_agent** で設計ツールを使った場合: **ER図作成中** / **フロー図作成中** / **画面項目定義書作成中** など、作業内容に応じたメッセージ

例（planning_agent のターミナル）:
```
[planning_agent] 依頼を受信、処理中...
[planning_agent] 応答送信開始
```

例（architect_agent が ER 図を書いたとき）:
```
[architect_agent] 依頼を受信、処理中...
[architect_agent] ER図作成中
[architect_agent] 応答送信開始
```

### 起動コマンド一覧を表示する

**プロジェクトルートで** 以下を実行すると、discovery の並び順に合わせた「各ターミナル用のコマンド」が表示されます。そのままコピーして各ターミナルで実行できます。

```bash
uv run python -m a4a.print_terminals
```

- **PowerShell（Windows）** と **bash** の両方のコマンドが出力されます。
- PowerShell のみ: `uv run python -m a4a.print_terminals --powershell`
- bash のみ: `uv run python -m a4a.print_terminals --bash`

### 手順

1. **ポート番号**は `discovery.py` の並び順で 8001 から振られます。  
   並びは `Path.iterdir()` に依存するため、**必ず `print_terminals` で表示されたコマンドを使う**か、同じディレクトリ構成で実行してください。

2. 各ターミナルで、**プロジェクトルートを cwd にしたうえで**、表示されたコマンドを実行します。

   **例（bash / Linux・Mac）**
   ```bash
   cd /path/to/A4A
   PORT=8000 uv run python -m a4a.agent
   ```
   ```bash
   PORT=8001 uv run python -m architect_agent.a2a_agent
   ```

   **例（PowerShell / Windows）**
   ```powershell
   cd C:\path\to\A4A
   $env:PORT="8000"; uv run python -m a4a.agent
   ```
   ```powershell
   $env:PORT="8001"; uv run python -m architect_agent.a2a_agent
   ```

3. すべて起動したら、ユーザーは **8000（Coordinator）** にだけクエリを送ります。

### メリット

- エージェントごとにターミナルが分かれるので、**どのエージェントのログか一目で分かる**。
- 特定のエージェントだけ再起動して挙動を確認しやすい。

### デメリット

- 起動するターミナル数が多くなる。
- ポートを手動で振るため、`discovery` の順序と合わせる必要がある（ずれると Coordinator が接続先を見誤る）。

## 3. Web UI で「どのエージェントが応答したか」と作業メトリクスを見る

`make ui` で Web UI（http://localhost:8888）を起動し、そこでクエリを送ると、次が表示されます。

- **ストリーミング応答の先頭**: どのエージェントが処理したか（`[AGENT:coordinator_agent]` → `[AGENT:○○_agent]`）
- **作業中（リアルタイム）**: 依頼を受けた直後に **「[作業中] エージェント名: リクエストを処理しています...」**、ツール実行時は **「[作業中] エージェント名: 作業内容」**（例: `[作業中] architect_agent: ER図作成`）、ツールを使わない応答の場合は **「[作業中] エージェント名: 応答を生成しています...」** が流れます。
- **応答の末尾**: **「[エージェント活動] エージェント名: 作業1, 作業2, 応答生成」** の形式で、そのエージェントが行った作業の一覧（作業報告・メトリクス）

設計ツール（write_design_doc）を使った場合は、doc_type に応じて「ER図作成」「フロー図作成」「画面項目定義書作成」などと表示されます。

**Web UI を起動したターミナル**（`uv run python -m a4a.web` を実行しているターミナル）にも、上記と同じ内容がリアルタイムで出力されます（`[ルーティング] → エージェント名`、`[作業中]`、`[エージェント活動]`）。すべての依頼で「どのエージェントがどんな作業をしているか」を UI とターミナルの両方で確認できます。

※ Web UI は「ルーティング先 1 エージェント」の応答を表示する構成です。Coordinator が内部で複数エージェントを呼ぶ場合は、表示されるのは主にルーティング先の 1 つです。

## 4. 方法の比較

| 方法 | 確認できること | 使いどころ |
|------|----------------|------------|
| **make run-verbose** | 全エージェントの uvicorn ログ（リクエスト・エラー）をプレフィックス付きでリアルタイム表示 | どのエージェントにリクエストが飛んでいるか・エラーが出ていないかを一画面で見たいとき |
| **エージェントごと別ターミナル** | 各エージェントのログを完全に分離 | 特定エージェントの挙動を詳しく追いたいとき |
| **Web UI** | 各クエリがどのエージェントにルーティングされたか | ブラウザで試しながら「今どれが動いたか」を確認したいとき |

## 5. 「このリクエストに誰が反応したか」を確認する

### 反応の形跡は残るか

**永続的な形跡（ログファイル・DB）は残りません。**

- **Web UI** (`make ui` → http://localhost:8888): ストリーミング応答の先頭に `[AGENT:coordinator_agent]` → `[AGENT:○○_agent]` がその場で表示されるだけです。誰が選ばれたかは**そのリクエストの画面上でしか分かりません**。
- **Coordinator 直接** (`make query` や `adk web` のチャット): Coordinator が内部でどのサブエージェントを呼んだかは、**コード上でログ出力していないため、後から「この発言に誰が反応した」と追うことはできません**。
- **run_all --verbose**: ターミナルに `[エージェント名:ポート]` 付きで uvicorn のアクセスログが出るため、**そのターミナルを見ていれば「いまどのエージェントにリクエストが飛んだか」は分かります**。ただしスクロールすると流れてしまうので、事後検証用の証跡としては弱いです。

### どのエージェントがどう反応するか

| 利用方法 | 動き |
|----------|------|
| **Web UI** | 1 クエリにつき **1 エージェント**だけが選ばれます。Gemini が「ユーザーの質問に最適なエージェント」をエージェント一覧（各 description）から 1 つ選び、そのエージェントが応答します。 |
| **Coordinator（adk web / make query）** | **PM 役の Coordinator** が受け取り、必要に応じて **複数エージェント** に順次依頼します。要望がふわっとしている場合はまず **planning_agent** に相談→具体化→**architect_agent**（設計）→**engineer_agent**（実装）の流れになるよう instruction で指定されています。 |

例:「ソースコードに合わせて README を作成するツールを作りたい。README には環境構築手順と何のためのどんなシステムかを書きたい」

- **Web UI** で送った場合: 「設計・ドキュメント」「実装」「企画」などの description に引っかかり、**architect_agent / engineer_agent / planning_agent** のいずれか 1 つが選ばれて応答する想定です（LLM の選択のため毎回同じとは限りません）。
- **Coordinator** に送った場合: 要望が抽象的と判断されれば、まず **planning_agent** に「何を決めるべきか・どう具体化するか」を相談し、そのあと **architect_agent** に設計（README の構成・成果物の置き場など）、**engineer_agent** に実装（ツール本体）を依頼する流れになり得ます。

### チャット応答末尾の「どのエージェントが何をしたか」

- **Web UI**: 応答の最後に「[エージェント活動] ○○_agent が応答しました。」が自動で付きます（ルーティング先の 1 エージェント名）。
- **Coordinator**（adk web や make query）: Coordinator の instruction により、応答の最後に「[エージェント活動] エージェント名: 実施したこと, ...」を 1 行で記載するよう指示しています。例: `[エージェント活動] planning_agent: 企画整理, architect_agent: ER図作成, engineer_agent: 実装`

## 6. さらに詳しく追いたい場合（ログレベル・ファイル出力）

- **uvicorn のログレベル**: 各 `a2a_agent.py` で `uvicorn.run(..., log_level="debug")` にすると、より詳細なログが出ます（本番では負荷に注意）。
- **ファイルに残す**: 標準出力をファイルにリダイレクトして後から確認する方法です。
  ```bash
  python -m a4a.run_all --verbose 2>&1 | tee agents.log
  ```
  `agents.log` に全エージェントのプレフィックス付きログが残ります。
